# Multi-App Support Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make a single Substrukt instance host N independent apps, each with its own schemas, content, uploads, version history, deployments, and API tokens.

**Architecture:** New `apps` and `app_access` tables in substrukt.db. Custom `AppContext` / `ApiAppContext` Axum extractors read `{app_slug}` from paths and enforce access. All content/schema/upload/deployment routes move under `/apps/{app_slug}/`. Cache keys gain app prefix (`{app_slug}/{schema_slug}/{entry_id}`). A startup file migration moves old single-app layout into `data/default/`. Backups remain instance-wide and need no changes.

**Tech Stack:** Rust, Axum, sqlx (SQLite), minijinja, htmx, DashMap

**Spec:** `docs/superpowers/specs/2026-03-30-multi-app-support-design.md` -- contains full SQL, type definitions, router structure, and edge case analysis. Reference it for details not repeated here.

    1. Extract `app_slug` from path params via `Path::<std::collections::HashMap<String, String>>::from_request_parts(parts, state)`. Get `"app_slug"` key.

    5. Return `AppContext { app }`.
  - Add helper methods:

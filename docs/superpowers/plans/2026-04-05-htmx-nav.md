# HTMX Navigation & Sidebar Fixes Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix three interconnected HTMX/navigation UX gaps: (1) sidebar not updating when navigating between apps via htmx, (2) flash messages lost during direct htmx navigation (non-redirect), (3) no active-state highlighting in sidebar links.

**Architecture:** The root cause of all three issues is that `hx-boost="true"` on `<body>` targets `#main-content` with `innerHTML` swap. This means the sidebar (`_nav.html`, included only by `base.html`) is never re-rendered during htmx navigation. The fix uses htmx's out-of-band swap (`hx-swap-oob`) mechanism: every htmx response includes an updated sidebar fragment appended after the main content, which htmx swaps into place alongside the main content. The sidebar gets an `id` attribute so htmx can target it for OOB replacement. Active state is determined client-side by comparing each link's `href` against `window.location.pathname` after each swap.

**Tech Stack:** Rust, Axum, minijinja, htmx (v2, hx-swap-oob), twind

## Design Decisions

### Sidebar update mechanism: OOB swap via `_partial.html`

- **Full-page reload** (changing `hx-boost` to target the entire page): Would work but defeats the purpose of htmx partial rendering. Rejected.
- **Client-side JS only** (parse URL, show/hide sidebar sections): Would handle active state but cannot update sidebar content (e.g., schema list changes when switching apps). Rejected for sidebar content updates.
- **OOB swap** (chosen): `_partial.html` already renders the main content for htmx requests. By appending a `<nav id="sidebar-nav" hx-swap-oob="true">...</nav>` fragment after the main content, htmx automatically finds it by `id` and replaces the existing sidebar. This is the canonical htmx approach for updating multiple page regions in one response.

To avoid duplicating nav markup, `_nav.html` is restructured to contain only the inner sidebar content (no wrapping `<nav>` tag). The `<nav>` wrapper lives in two places: `base.html` (full page loads) and `_partial.html` (OOB swap on htmx responses). `_nav.html` remains the single source of truth for sidebar content.

### Active state: client-side JavaScript

A JS function `highlightActiveNav()` runs on `DOMContentLoaded` and `htmx:afterSettle` (after URL is updated by `hx-push-url`). It compares each sidebar link's `href` against `window.location.pathname` and toggles CSS classes. This avoids modifying 20+ route handlers to pass `current_path` to template context.

### Flash messages: no changes needed

Flash messages work correctly in all navigation flows:
- **Post-redirect-get (non-htmx):** Flash set, redirect happens, full page load consumes flash via `base.html`. Works.
- **Post-redirect via htmx-boosted form:** `htmx_aware_redirect` in `src/auth/mod.rs` sends `HX-Redirect` header, causing a full page load. Flash renders via `base.html`. Works.
- **Direct htmx GET navigation:** `_partial.html` already renders `{% if flash_kind %}...{% endif %}` (lines 2-8). Any pending flash is consumed by `take_flash()` and rendered inline. Works.

The OOB sidebar swap from Task 1 does not interfere with flash rendering because the flash div is inside the main content block, and the OOB nav is a separate fragment outside it.

## Task 2: Add active-state highlighting to sidebar navigation

**Files:** `templates/base.html`

**Depends on:** Task 1

- [ ] **Step 1:** Add a JavaScript function `highlightActiveNav()` in `templates/base.html` inside the existing `<script>` block, before the `initMarkdownEditors()` call (line 230). This function:
  1. Gets the current path from `window.location.pathname`.
  2. Queries all `#sidebar-nav a` elements.
  3. For each link, checks if the current path matches the link's `href` (exact match for `/apps`, prefix match for all other links).
  4. Adds/removes CSS classes for the active state.

  ```javascript
  function highlightActiveNav() {
    var path = window.location.pathname;
    document.querySelectorAll('#sidebar-nav a').forEach(function(link) {
      var href = link.getAttribute('href');
      if (!href) return;
      var isActive;
      if (href === '/apps') {
        isActive = (path === '/apps');
      } else {
        isActive = (path === href) || path.startsWith(href + '/');
      }
      if (isActive) {
        link.classList.add('bg-sidebar-hover', 'text-white');
      } else {
        link.classList.remove('bg-sidebar-hover', 'text-white');
      }
    });
  }
  ```

- [ ] **Step 2:** Call `highlightActiveNav()` at the end of the existing initialization block (after `initSubmitSpinner();` at line 232):
  ```javascript
  highlightActiveNav();
  ```

- [ ] **Step 3:** Add a `htmx:afterSettle` event listener after the existing `htmx:afterSwap` handler (after line 243). We use `afterSettle` rather than `afterSwap` because the URL may not be updated yet during `afterSwap`:
  ```javascript
  document.body.addEventListener('htmx:afterSettle', function() {
    highlightActiveNav();
  });
  ```

  The existing `htmx:afterSwap` handler (lines 233-243) for markdown editors, upload zones, spinners, and page title remains unchanged.

## Task 3: Fix missing nav variables in handler contexts

**Files:** `src/routes/settings.rs`, `src/routes/mod.rs`

**Depends on:** Task 1 (because OOB sidebar swap requires all handlers to pass nav variables)

After Task 1, every htmx partial response includes `_nav.html` via OOB swap. The nav template uses `user_role`, `csrf_token`, `current_username`, `app`, and `nav_schemas`. If a handler does not pass these variables, minijinja will treat them as undefined (falsy/empty), causing the sidebar to render incorrectly -- admin-only links will be hidden, the CSRF token in the logout form will be empty, and the username display will be blank.

The following handlers are missing required nav variables:

| Handler | File | Missing variable(s) |
|---------|------|---------------------|
| `invite_user` | `src/routes/settings.rs` (line 178 context) | `user_role` |
| `render_users_with_error` | `src/routes/settings.rs` (line 222 context) | `user_role` |
| `audit_log_page` | `src/routes/settings.rs` (line 355 context) | `csrf_token` |
| `render_error` | `src/routes/mod.rs` (line 117 context) | `user_role`, `current_username`, `csrf_token` |
| `not_found` | `src/routes/mod.rs` (calls `render_error`) | Needs to pass session data to `render_error` |

Non-app-scoped pages (`/apps`, `/settings/*`) do not pass `app` or `nav_schemas` -- this is correct because the `{% if app %}` block in `_nav.html` simply won't render, showing only global links.

# User Management Improvements Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix two user management UX gaps: (1) The Users page only shows the invite form and pending invitations -- it does not list registered users. Add a table of all users with their roles. (2) There is no way for a user to change their password. Add a `/settings/profile` page accessible to all authenticated users with a password change form.

**Architecture:** New DB query functions in `models.rs` (`list_users`, `update_user_password`, `find_user_by_id`). New route handlers in `settings.rs` for the profile page. Updated `users.html` template to include a registered users table. New `profile.html` template. Nav link for profile page visible to all users. Admin-only user role changes from the users list are out of scope for this milestone (just display, not edit).

**Tech Stack:** Rust, Axum, SQLite (sqlx), minijinja templates, htmx, twind CSS

  ```rust
      let users = sqlx::query_as::<_, User>("SELECT * FROM users ORDER BY username")
          .fetch_all(pool)
          .await?;

  }
  ```

  ```rust
      let user = sqlx::query_as::<_, User>("SELECT * FROM users WHERE id = ?")
          .bind(user_id)
          .fetch_optional(pool)
          .await?;
      Ok(user)
  }
  ```

- [ ] **Step 3: Add `update_user_password` function**

  Add after `find_user_by_id`. This takes a user_id and new plaintext password, hashes it, and updates the row:

  ```rust
  pub async fn update_user_password(pool: &SqlitePool, user_id: i64, new_password: &str) -> eyre::Result<()> {
      let password_hash = User::hash_password(new_password)?;
      sqlx::query("UPDATE users SET password_hash = ? WHERE id = ?")
          .bind(&password_hash)
          .bind(user_id)
          .execute(pool)
          .await?;
      Ok(())
  }
  ```

- [ ] **Step 1: Add the `profile_page` handler**

  In `src/routes/settings.rs`, add a new handler function. This page is available to all authenticated users (not admin-only), so it does NOT call `auth::require_role`. The `require_auth` middleware already ensures the user is logged in. It shows the current username and a password change form:

  ```rust
  async fn profile_page(
      HxRequest(is_htmx): HxRequest,
      State(state): State<AppState>,
      session: Session,
  ) -> axum::response::Result<axum::response::Response> {
      let user_id = auth::current_user_id(&session)
          .await
          .ok_or("Not authenticated")?;

      let user = models::find_user_by_id(&state.pool, user_id)
          .await
          .map_err(|e| format!("DB error: {e}"))?
          .ok_or("User not found")?;

      let csrf_token = auth::ensure_csrf_token(&session).await;
      let (flash_kind, flash_message) = match auth::take_flash(&session).await {
          Some((kind, msg)) => (kind, msg),
          None => (String::new(), String::new()),
      };
      let user_role = auth::current_user_role(&session).await.unwrap_or_default();
      let current_username = auth::current_username(&session).await.unwrap_or_default();

      let tmpl = state
          .templates
          .acquire_env()
          .map_err(|e| format!("Template env error: {e}"))?;
      let template = tmpl
          .get_template("settings/profile.html")
          .map_err(|e| format!("Template error: {e}"))?;
      let html = template
          .render(minijinja::context! {
              base_template => base_for_htmx(is_htmx),
              csrf_token => csrf_token,
              user_role => user_role,
              current_username => current_username,
              username => user.username,
              flash_kind => flash_kind,
              flash_message => flash_message,
          })
          .map_err(|e| format!("Render error: {e}"))?;
      Ok(Html(html).into_response())
  }
  ```

- [ ] **Step 2: Add the `change_password` handler**

  Add a form struct and handler:

  ```rust
  #[derive(serde::Deserialize)]
  struct ChangePasswordForm {
      current_password: String,
      new_password: String,
      confirm_password: String,
  }

  async fn change_password(
      State(state): State<AppState>,
      session: Session,
      Form(form): Form<ChangePasswordForm>,
  ) -> axum::response::Result<axum::response::Redirect> {
      let user_id = auth::current_user_id(&session)
          .await
          .ok_or("Not authenticated")?;

      // Validate new password length
      if form.new_password.len() < 8 {
          auth::set_flash(&session, "error", "New password must be at least 8 characters").await;
          return Ok(axum::response::Redirect::to("/settings/profile"));
      }

      // Confirm passwords match
      if form.new_password != form.confirm_password {
          auth::set_flash(&session, "error", "New passwords do not match").await;
          return Ok(axum::response::Redirect::to("/settings/profile"));
      }

      // Verify current password
      let user = models::find_user_by_id(&state.pool, user_id)
          .await
          .map_err(|e| format!("DB error: {e}"))?
          .ok_or("User not found")?;

      if !user.verify_password(&form.current_password) {
          auth::set_flash(&session, "error", "Current password is incorrect").await;
          return Ok(axum::response::Redirect::to("/settings/profile"));
      }

      // Update password
      models::update_user_password(&state.pool, user_id, &form.new_password)
          .await
          .map_err(|e| format!("DB error: {e}"))?;

      state.audit.log(
          &user_id.to_string(),
          "password_changed",
          "user",
          &user_id.to_string(),
          None,
      );

      auth::set_flash(&session, "success", "Password updated successfully").await;
      Ok(axum::response::Redirect::to("/settings/profile"))
  }
  ```

## Design Decisions

1. **Profile page accessible to all roles, not just admin.** Every user should be able to change their own password. The `/settings/profile` route uses `current_user_id` (which just requires authentication) rather than `require_role("admin")`.

2. **Users list is read-only for now.** The registered users table displays username, role, and created date. Admin actions like changing a user's role or deleting a user are out of scope. This keeps the change minimal and focused on the two reported gaps.

3. **Password change requires current password.** Standard security practice. Prevents session hijacking from escalating to account takeover. The handler verifies the current password before updating.

4. **Password minimum 8 characters.** Enforced both server-side (handler validation) and client-side (HTML `minlength` attribute). This matches common security baseline without being overly restrictive.

5. **Flash messages for password change feedback.** Uses the existing `set_flash`/`take_flash` pattern (same as backups page) with POST-redirect-GET to avoid form resubmission. Success and error messages use consistent styling.

6. **Nav link via username text.** Rather than adding a separate "Profile" link that takes up nav space, the existing username text in the bottom nav area becomes a clickable link. This is discoverable (users naturally look at their name) and space-efficient.

7. **Audit logging for password changes.** The `change_password` handler logs a `password_changed` audit event with the user's ID. No details are included (no password data in logs).
